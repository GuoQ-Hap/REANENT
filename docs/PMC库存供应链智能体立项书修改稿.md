# 凌昂科技 PMC 库存供应链智能体项目立项书修改稿



## 一、项目定位

本项目拟建设面向跨境电商 PMC 场景的库存供应链智能体，建议产品名为“PMC 库存计划官”或“小昂库存计划官”。

项目目标是以“结构化数据 + 规则计算 + 预测补货模型 + Agent 编排 + 人工确认闭环”为核心，建设一个可信、可解释、可复算、可审计的库存计划辅助系统。

系统首期定位为 PMC 日常库存控制与供应链协同工具，重点解决以下问题：

- 库存风险发现不及时，断货、冗余、在途异常、采购异常分散在多张表和多个系统中。
- 发货、采购、补货建议依赖人工经验，计算口径复杂，复盘困难。
- 缺少对 SKU / MSKU / FNSKU 的全链路库存解释能力，业务人员难以快速追因。
- 大模型可以理解和解释业务问题，但不能直接承担数量、日期、金额等高风险计算。
- 当前建议、人工修改、确认、关闭和复盘缺少统一记录，无法形成持续优化闭环。

## 二、建设原则

1. 大模型负责理解、编排、解释和追问，不直接编造库存、采购、发货、金额、日期等结果。
2. 数量、日期、金额、风险等级、补货量由规则工具和预测模型计算，保证可复算。
3. 高风险动作必须人工确认，首期只生成建议、草稿和解释，不自动创建采购单、发货单或写回 ERP / WMS。
4. 每个建议都必须展示数据来源、计算口径、原因链、风险等级和人工确认点。
5. 所有模型输入输出、状态流转、工具调用、人工修改和反馈结果都必须记录。
6. 架构按模块解耦，数据接入、确定性计算、Agent 编排、知识治理、业务应用可独立测试和演进。

## 三、总体技术架构

参考第二张架构图，系统建议分为六层：

```text
业务应用层
  -> 知识与治理层
  -> Agent 编排层
  -> 确定性计算层
  -> 数据底座层
  -> 数据接入层
  -> 反馈评估闭环
```

### 3.1 业务应用层

业务应用层面向 PMC、采购、物流、运营和管理层，提供可操作的业务入口。

首期应用模块包括：

- 库存控制塔：展示断货风险、冗余库存、库存异常、物流异常、采购异常。
- 断货追因：按 SKU / MSKU / FNSKU 输出库存、需求、在途、采购、发货、物流等原因链。
- 发货验证：复算基础发货量、修正量、参考量，解释差异来源。
- 采购验证：复算基础采购量、修正量、参考量，处理 MOQ、箱规、拼单、需求过低等规则。
- 周度发货计划：生成周度发货计划草稿，包含渠道、仓库、箱数、待配货和异常原因。
- AI 预测补货：结合销量预测、库存仿真、供货周期和补货策略，输出补货建议。
- 异常 Case：生成异常原因、处理建议、责任角色、状态和反馈记录。
- Agent 问答：解释字段、规则、建议原因、计算路径和风险结论。

### 3.2 知识与治理层

知识与治理层为 Agent 和规则工具提供业务约束。

包括：

- RAG 知识库：字段说明、SOP、备货规则、发货规则、采购规则、库存口径、异常处理规范。
- 规则版本库：安全库存、Lead Time、MOQ、箱规、补货频率、发货周期、采购周期等版本。
- 权限控制：按角色、店铺、国家、销售部门、负责人控制数据可见范围。
- 审计留痕：记录建议生成、模型输入输出、工具调用、人工修改、导出、确认、关闭。
- 人工确认：采购建议、发货计划、异常 Case、高风险规则变更必须人工确认。

### 3.3 Agent 编排层

Agent 编排层是系统的大脑，但不直接替代规则工具。

核心职责：

- 理解用户问题，判断是库存风险、断货追因、发货验证、采购验证、周度计划、规则问答还是异常 Case。
- 根据任务类型选择工具和执行路线。
- 在工具返回后继续判断是否需要补充数据、追问用户、生成解释或进入人工确认。
- 将结构化结果转换成业务可读输出。
- 在异常、缺字段、无数据、模型不确定时主动降级和提示。

Agent 编排层可分为以下大模型模块：

| 模块 | 作用 | 输入 | 输出 |
| --- | --- | --- | --- |
| 意图识别模型 | 判断用户目标和任务类型 | 用户问题、上下文、可选任务类型 | task_type、confidence、风险和数据需求 |
| 上下文判断模型 | 判断是否需要历史对话或业务上下文 | 当前问题、最近对话 | 是否需要上下文、需要哪些上下文 |
| 工具编排模型 | 选择下一步工具或追问 | 用户目标、工具列表、已获得观察结果 | 下一步 action、参数、原因 |
| 失败处理模型 | 处理数据库、工具、字段缺失等失败 | 错误信息、任务状态 | 用户提示、重试/降级建议 |
| 业务解释模型 | 解释原因链、建议逻辑和人工确认点 | 规则结果、证据、风险等级 | 业务说明、注意事项 |
| 总结模型 | 压缩长链路观察和历史执行结果 | 多轮工具结果、状态历史 | 摘要和下一步建议 |

### 3.4 确定性计算层

确定性计算层负责所有可复算结果，不能由大模型替代。

首期工具包括：

- 库存健康计算：断货预警、冗余库存、库存异常、库龄风险。
- 可售天数 / Lead Time：FBA 可售、本地仓、海外仓、在途、采购周期、生产周期、头程运输、清关、入仓预约。
- 发货复算：基础发货量、发货修正量、发货参考量、待配货量。
- 采购复算：基础采购量、采购修正量、采购参考量、MOQ、箱规、拼单、需求过低。
- 库存核对：可用库存、在途库存、预留库存、缺口量、超发、覆盖天数。
- 补货预测：销量预测、库存仿真、补货点、建议补货量、预计缺货风险。
- 物流建议：慢船、美森、空运/海运、暂停发货、延迟、在途关注。

### 3.5 数据底座层

数据底座建议以 Doris 数据仓库 `dw_leang` 为核心，形成统一读取层。

首期主要表池：

| 表 | 用途 |
| --- | --- |
| `ads_lingxing_all_warehouse_new_v1` | 库存控制塔主宽表，适合做月度基准快照和主入口 |
| `ads_lingxing_sc_sales_daily_new` | 销售日报、真实销量、广告和促销数据 |
| `ads_lingxing_stocking_rules` | ADS 层备货规则 |
| `dim_inventory_forecast_v1` | 采购侧未来库存预测 |
| `dim_inventory_forecast_v1_fh` | 发货侧未来库存预测 |
| `dwd_lingxing_fba_warehouse_detail` | FBA 明细库存 |
| `dwd_lingxing_inventory_details` | 国内仓库存明细 |
| `temp_lingxing_stocking_rules` | 临时/现行备货规则 |

注意：`ads_lingxing_all_warehouse_new_v1` 更新频率较低，应作为主查询宽表和基准快照，不应单独承担实时库存、采购在途、发货在途追踪。涉及“今天库存”“最新在途”“最新发货”的问题，必须结合 DWD / ODS 明细、预测表和实时快照表复核。

### 3.6 数据接入层

数据接入层负责统一接入：

- 领星 ERP
- Amazon FBA
- 国内仓 / WMS
- 采购物流数据
- 备货规则 Excel
- 销售 / 广告 / 促销数据
- 外部节假日、旺季、竞品、市场因素数据

接入后的数据统一进入数据仓库或只读 connector，Agent 只能通过白名单工具查询，不直接拼 SQL 或越权访问。

## 四、AI 预测补货模型设计

参考第一张“AI 预测补货模型”图，本项目建议新增“预测补货”能力，作为库存控制塔、采购验证和周度计划的上游输入。

### 4.1 数据来源

预测补货模型需要融合以下数据：

- Amazon 销售数据：销量、订单、转化、退货。
- 广告数据：SP / SB / SD 投放、广告花费、广告销量。
- 库存数据：FBA、海外仓、国内仓、在途、预留、待配货。
- 供应链数据：采购单、生产周期、物流周期、清关、入仓预约。
- 产品数据：类目、价格、生命周期、新品/老品、套装关系。
- 促销活动数据：Coupon、Deal、秒杀、站内外活动。
- 外部数据：节假日、旺季、季节性、竞品变化、平台政策。

### 4.2 数据处理与特征工程

处理流程：

1. 数据清洗：去重、补全、异常值处理。
2. 数据整合：按 SKU / MSKU / FNSKU、站点、店铺、日期统一粒度。
3. 特征工程：
   - 历史销量特征：近 7 / 14 / 30 / 60 / 90 天销量。
   - 价格与广告特征：价格变动、广告花费、广告销量、ACOS。
   - 库存特征：可售库存、预留、在途、可用库存、库龄。
   - 时间特征：星期、月份、节假日、旺季、淡季。
   - 商品特征：新品、老品、生命周期、类目、套装。

### 4.3 销量预测模型

采用多模型融合，而不是单一模型。

可选模型：

- 统计模型：移动平均、指数平滑、ARIMA、Prophet。
- 机器学习模型：LightGBM、XGBoost、Random Forest。
- 场景修正模型：新品冷启动、促销放量、断货恢复、旺季备货、广告投放变化。

输出建议：

- 未来 7 / 14 / 30 / 60 / 90 天销量预测。
- P25 / P50 / P75 / P90 多分位预测。
- 预测置信区间。
- 预测偏差监控。

### 4.4 库存仿真与风险评估

库存仿真应结合销量预测和供应链周期。

核心变量：

- FBA 可售库存
- FBA 预留库存
- 在途库存
- 海外仓库存
- 国内仓库存
- 待生产库存
- 采购周期
- 生产周期
- 质检入库周期
- 头程运输周期
- 清关周期
- 入仓预约周期

核心输出：

- 未来库存曲线。
- 安全库存线。
- 预计缺货时间点。
- 可售天数。
- 断货概率。
- 冗余风险。
- 预测风险。
- 库存周转天数。

### 4.5 补货决策引擎

补货决策引擎按 SKU 分层输出策略。

建议分层：

| SKU 类型 | 策略 |
| --- | --- |
| 爆品 / A 类 | 防断货优先，允许更高安全库存 |
| 稳定款 / B 类 | 平衡库存与资金占用 |
| 长尾款 / C 类 | 防滞销优先，低频补货 |
| 新品 | 小批量多频次，结合新品爬坡曲线 |
| 清仓款 | 原则上不补货，只消化库存 |

补货计算建议口径：

```text
补货点 = 预测销量(周期) + 安全库存
安全库存 = Z 值 * 需求波动 * sqrt(周期)
建议补货量 = 目标库存水位 - 当前有效库存
目标库存水位 = 目标覆盖天数 * 日均预测销量
当前有效库存 = FBA 可售 + 海外仓可用 + 国内仓可用 + 在途可用 - 已锁定/待分配
```

输出字段：

- 建议补货数量
- 建议下单时间
- 建议运输方式
- 补货优先级
- 预计资金占用
- 预计缺货损失
- 滞销风险
- 人工确认点

## 五、Agent 业务流程设计

### 5.1 库存控制塔流程

```text
读取主宽表与明细表
  -> 计算库存健康指标
  -> 识别断货、冗余、在途、采购、物流风险
  -> 按风险等级、店铺、国家、PMC、MSKU、FNSKU 分组
  -> 生成风险列表
  -> Agent 解释原因链和建议动作
  -> 人工确认或创建异常 Case
```

### 5.2 单 SKU 断货追因流程

```text
用户输入 SKU / MSKU / FNSKU
  -> Agent 判断任务类型
  -> 查询 FBA、国内仓、海外仓、在途、采购、发货、预测数据
  -> 规则工具计算缺口量和预计断货时间
  -> 生成原因链
  -> 给出发货、采购、调拨、物流建议
  -> 记录人工反馈
```

### 5.3 发货验证流程

```text
读取发货侧预测表和库存明细
  -> 复算基础发货量
  -> 计算发货修正量和参考量
  -> 校验库存是否足够、是否超发、是否覆盖在途
  -> 输出差异解释
  -> 人工确认后进入周度发货计划草稿
```

### 5.4 采购验证流程

```text
读取采购侧预测表、备货规则、MOQ、箱规和供应商周期
  -> 复算基础采购量
  -> 计算采购修正量和参考量
  -> 处理 MOQ、半箱、整箱、可拼采购、需求过低规则
  -> 输出采购建议草稿
  -> 人工确认，不自动创建正式采购单
```

### 5.5 周度发货计划流程

```text
汇总库存风险、补货预测、发货验证和物流约束
  -> 生成周度发货计划草稿
  -> 展示发货时间、渠道、仓库、箱规、待配货、异常原因
  -> PMC / 物流 / 运营确认
  -> 记录修改原因和确认结果
```

## 六、可视化与交互设计

### 6.1 控制塔看板

核心指标：

- 高风险断货 SKU 数
- 冗余库存 SKU 数
- 可售风险 SKU 数
- 建议补货金额
- 预计缺货损失
- 预计释放库存金额

图表建议：

- 库存健康分布
- 可售天数分布
- 断货风险 Top SKU
- 未来库存曲线
- 预测销量曲线
- 补货建议列表

### 6.2 Agent 解释区

Agent 输出必须包含：

- 结论
- 关键数据
- 计算口径
- 原因链
- 建议动作
- 人工确认点
- 数据来源

示例：

```text
SKU A001 在美国站预计 18 天后断货。
主要原因：
1. 最近 14 天销量较 30 天均值增长 42%。
2. 广告预算提升后转化率同步提升。
3. 当前 FBA 可售库存仅可覆盖 18 天。
4. 最近一批在途预计 28 天后入仓。

建议：
立即安排空运 800 件，同时海运补 3000 件。
```

## 七、反馈评估闭环

系统需要建立持续评估机制。

评估指标：

- 建议采纳率
- 命中率
- 预测偏差
- 复算一致率
- 人工修改率
- 缺货减少率
- 冗余库存下降金额
- 发货计划准时率
- 采购建议准确率

反馈数据应回流到：

- 预测补货模型
- 风险阈值
- 备货规则
- Agent 提示词
- 工具编排策略
- 知识库和 SOP

## 八、首期范围调整建议

在第一版基础上，建议首期范围调整为“控制塔 + 断货追因 + 发货/采购复算 + 预测补货基础版 + Agent 解释”。

### 8.1 首期必须完成

- 库存控制塔基础看板。
- 单 SKU / FNSKU 库存风险查询。
- 断货原因链。
- 发货验证复算。
- 采购验证复算。
- 预测补货基础版：未来销量预测、库存仿真、补货点和建议补货量。
- Agent 解释与规则问答。
- 模型输入输出日志。
- 状态流转日志。
- 人工确认记录。

### 8.2 首期不做或只做草稿

- 不自动创建采购单。
- 不自动修改发货单。
- 不自动写回库存或订单。
- 不自动修改备货规则。
- 不做完整供应商协同平台。
- 不做财务级现金流优化。
- 不做复杂多级库存优化。

## 十、验收标准

### 10.1 功能验收

- 能展示库存控制塔和风险列表。
- 能按 SKU / MSKU / FNSKU 查询库存风险。
- 能输出断货原因链。
- 能生成发货验证表并解释差异。
- 能生成采购验证表并解释差异。
- 能输出预测补货建议和未来库存曲线。
- 能生成周度发货计划草稿。
- 能回答字段、规则和建议原因问题。
- 能记录人工确认和处理结果。

### 10.2 可信验收

- 每个数量、日期、金额都有计算来源。
- 每个建议都能追溯到数据表、字段和规则。
- 大模型输入输出可查。
- 状态流转可查。
- 高风险动作不会自动执行。
- 数据缺失时必须提示缺失原因，不允许用演示数据或模型猜测替代。

### 10.3 测试验收

- 每个模块有独立测试。
- 每个模型模块支持独立调试输入。
- 每个工具有输入输出契约测试。
- 每个业务场景有状态流转联动测试。
- 每个模块输出分级日志。
- 模型交互记录按类型和时间 ID 落盘。

## 十一、主要风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 数据口径不一致 | 主宽表、明细表、预测表结果不一致 | 定义主路径和校验路径，输出数据来源和字段口径 |
| 预测偏差 | 销量预测不准导致补货建议偏差 | 建立预测偏差监控，引入人工修正和分位预测 |
| 规则复杂 | 采购、发货、箱规、MOQ、拼单规则多 | 高频规则先固化，低频规则进入人工确认 |
| 大模型幻觉 | 编造库存、日期、金额 | 模型只解释和编排，计算由工具 |
| 自动化风险 | 错误创建采购或发货动作 | 首期只生成草稿，不写回正式系统 |
| 业务不信任 | 建议看不懂、不可复盘 | 输出原因链、证据、公式、数据来源和人工确认点 |
| 系统耦合重 | 多系统、多表、多规则接入慢 | 先接只读 connector 和核心表池，再逐步扩展 |

## 十二、本次修改相较第一版的重点变化

1. 新增“AI 预测补货模型”章节，将销量预测、库存仿真、补货策略和补货建议纳入项目主线。
2. 将技术架构明确为“数据接入层 - 数据底座层 - 确定性计算层 - Agent 编排层 - 知识与治理层 - 业务应用层 - 反馈评估闭环”。
3. 强化 Doris 数据仓库 `dw_leang` 和核心表池在系统中的位置。
4. 明确大模型只负责理解、编排、解释、追问，不直接计算数量、日期、金额。
5. 增加模型模块拆分，包括意图识别、上下文判断、工具编排、失败处理、业务解释和总结。
6. 将预测偏差、采纳率、命中率、复算一致率纳入反馈评估闭环。
7. 将“每个模块独立测试、日志分级、模型输入输出落盘、状态流转测试”作为验收要求。
8. 首期范围增加“预测补货基础版”，但仍坚持“不自动下单、不自动写回、不自动改规则”的风险边界。
